# Catalog (optional module)

> **Optional module.** Enable it only for subjects that genuinely have a **closed set of named,
> recurring solutions** the learner must choose between — named techniques, named forms, named
> strategies. Most subjects do not. Disable it and the core is untouched.

## When to Activate This Skill

- "Which of these applies here?" — choosing between named approaches
- Explaining one entry, or deciding which unit a concept belongs in
- Checking an entry is being used for the right reason, not by reflex

## When a catalog earns its place

A catalog is worth maintaining when **all** of these hold:

1. The subject has a **bounded, named** set of recurring solutions (not an open-ended space).
2. Learners must **choose between them**, and the wrong choice is a real, common mistake.
3. The names are used consistently in the field, so the catalog matches what the learner will meet
   outside the course.

If the set is open-ended, or the entries are never in competition, a catalog is overhead — the
lessons themselves are the reference.

## Entry shape (keep every entry to these five lines)

- **Name** — as the field uses it.
- **Category** — the grouping the course teaches it under.
- **Intent** — the *change or variation it absorbs*, in one sentence. This is what makes it
  selectable.
- **Role in the running example** — the concrete place in this course's running example where it
  applies.
- **Common misuse** — the mistake learners actually make with it.

An entry without an **intent** and a **common misuse** is a glossary line, not a catalog entry.

## How to pick, and how to teach picking

Name the **change you want to absorb** — a new kind of input? a new variation in the process? a new
requirement that will keep shifting? — and that points to the entry. Teach the selection question,
not the list: a learner who has memorized the catalog but cannot name what varies has learned
nothing usable.

**If nothing varies, no entry is needed.** Say so explicitly. Applying a named solution for its own
sake is the most common mistake in any catalog-shaped subject, and the catalog itself is what
tempts it — so the catalog is also the place to warn against it.

## Keeping the catalog honest

- **One entry per idea.** If two entries have the same intent, merge them.
- **The catalog is a reference, not a syllabus.** Its order is for lookup; the teaching order comes
  from the syllabus (skill `backward-design`), sequenced by dependency.
- **Every entry appears in a lesson.** A catalog entry the course never teaches is scope creep.
- **Prune on review.** Entries that no lesson references and no exercise requires are removed, not
  kept for completeness.

## Provenance

Derived from the reference course's `design-pattern-catalog` skill — the per-entry shape (category ·
intent · role in the running example · common misuse), the "name the change you want to absorb"
selection heuristic, and the "if nothing varies you do not need one; applying it for its own sake is
the commonest mistake" warning — all generalized away from that course's subject-specific entry list
and language conventions, which were dropped with it. Kept as an **optional module** rather than
core on the critical-thinking filter: the shape is sound, but it presupposes a bounded named
solution set that most subjects do not have, and the research digest gives no support for a catalog
as a universal course component. See `CLASSIFICATION.md`.
